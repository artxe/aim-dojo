# calc/

Numeric sens/DPI/mouse-scale math. DOM lives in `controller/`, FOV side effects in `logic.js`, worker tuple format in `worker/`.

## Files
- `calc_sens.js` — per-game in-engine sens; `calc_rad_per_px()`; `lol_sens_to_dpi_scale()`; `calc_sens_r6_file()` (R6 `GameSettings.ini` values: yaw/pitch fixed at 1 with the 6dp `MouseSensitivityMultiplierUnit` carrying the whole hipfire sens, 8 ADS scope ints ≤200 on a shared 6dp `ADSMouseMultiplierUnit` grid-scanned over `[max_ratio/200, max_ratio/100]` for min worst-scope error — feeds BOTH the worker display tuple and the INI rewrite). Imports `../math.js` + immutable `../constants.js` only — NO `state`
- `calc_dpi.js` — per-game DPI to preserve turning; reads `state.dpi_norm`
- `calc_pubg.js` — PUBG integer ↔ yaw conversion

## Non-obvious
- `calc_rad_per_px(fov_deg, width)` returns rad/px. The monitor-match point is derived INTERNALLY from FOV geometry — `match = width * (1 - cos(fov/2)) / 2` — the sagitta (arc-height, in px) of the FOV arc taking `width` as the circle's diameter, NOT arc-height÷diameter (that ratio is just the `(1 - cos(fov/2)) / 2` factor) — so the matching distance grows super-linearly with FOV and collapses to 0 (plain linear sens) as FOV → 0. Because `match ∝ width`, the `width` cancels in the return and the result is exactly ∝ `1/width` — which is WHY `match` must stay a length (px), not the dimensionless ratio. The returned value is the mean rad/px over `[0, match]` (a secant slope), slightly below the on-center peak (~3% at 90°, ~18% at 120°). It's the hot inner of nearly every sens calc. `on_mousemove` does NOT call it directly — `state.camera.sens` is the cached result, recomputed in `logic.update_camera_view()` and `render.resize_3d()`
- There is NO `match`/DPI parameter (removed along with the worker `dpi_x` field). DPI affects sensitivity ONLY via (a) OS pixels feeding the cached `state.camera.sens` for the camera, and (b) the explicit `dpi * sens` factor inside `calc_dpi_*`. The matching point itself is purely geometric (FOV + width), identical across DPIs
- Two layers: `calc_dpi_*` (main thread) read everything from `state` and take ZERO params; `calc_sens_*` / `calc_rad_per_px` are PURE and shared with `worker/calc_worker.js` (which has no `state`), so they MUST stay parameterized (`fov`/`width`/`height` only) — do NOT make them read `state` to "reduce params" (it would break the worker)
- `calc_sens.js` may import ONLY `../math.js` + immutable `../constants.js` — enforced by a file-scoped `no-restricted-syntax` allow-list in `eslint.config.js` (NOT an inline header). This keeps the worker import tree (`calc_worker → calc_sens → math/constants`, `constants.js` imports nothing, and `math.js` itself bans all imports) closed so `state.js` can never transitively reach the worker. Same guard exists for `calc_worker.js` (allow `./calc_sens.js` + `../math.js` only)
- Aspect-ratio-aware games (`al`, `bdo`, `cs2`, `mc`, `r6`) feed through `convert_deg_across_aspect` first; aspect-independent (`fn`, `ow`, `pubg`, `val`) don't (`sa` is aspect-aware too, but hardcodes 4:3 via `height * 4 / 3` instead of calling `convert_deg_across_aspect`). Mirror this when adding a game. Two different reference bases show up within the aware group: `al`/`cs2` store a HORIZONTAL fov defined at a fixed 4:3 basis (`convert_deg_across_aspect(hfov, 4/3, width/height)` re-bases it to the real aspect), while `bdo`/`mc`/`r6` store a VERTICAL fov (`convert_deg_across_aspect(vfov, height, width)` derives the real-aspect horizontal from it) — check which basis a new game's engine actually uses before copying either pattern.
- `convert_deg_across_aspect` doubles as a fixed-ratio zoom when BOTH spans are constants (not screen `width`/`height`). All FOV values — base and zoomed — now live in `constants.js`'s `fov.<game>` IIFEs, computed ONCE at module load: `val`'s 2.5× ADS FOV is `convert_deg_across_aspect(hipfire, 2.5, 1)` (same constant-span pattern for `al`/`r6`'s `xN` fields), and `ow`'s hero FOVs (`ashe`/`emre`/`freja`/`widow`) also moved to a FIXED `convert_deg_across_aspect(angle, 9, 16)` — no longer the real screen `height`/`width` a pre-refactor worker call used. The `(angle, 9, 16)` span order isn't arbitrary: like `bdo`/`mc`/`r6`, OW's per-hero scope values are VERTICAL fov in-game, so `angle` is re-based from a 9-unit (vertical) span to a 16-unit (horizontal) one to get the horizontal fov `calc_sens_ow`/`render` need — `render`'s `widow` entry is deliberately the RAW (unconverted) constant for exactly this reason, since `render[]` wants vertical-at-16:9 already. `logic.update_camera_view()` and `calc_worker.js` just read the resulting `constants.fov.<game>.<field>`; neither calls the helper for zoom itself anymore. Constant-span calls are aspect-INDEPENDENT — they don't go stale on resize, so `val`/`ow` stay OUT of the aspect-dependent FOV list (`render/CLAUDE.md`) even though `constants.js` calls the helper for them
- `calc_sens_*` returns raw in-engine sens. The 52-value `calc_worker.update_game_sens` derives multipliers (e.g. `x2`, `x4`, scope) by re-calling at the zoomed FOV (read from `constants.fov.<game>.<field>`) and dividing by the hipfire baseline. That field is usually `convert_deg_across_aspect(base, N, 1)` (constant-span N× zoom: `al`/`r6`/`val` scopes) — NOT a literal `FOV/N`; only `pubg` divides the FOV directly (`base/N`), and `cs2` stores literal scope FOVs
- `lol_sens_to_dpi_scale()` lives here but is LOL-only — piecewise-linear over 11 anchors (`.031_25 … 3.5`). LOL has no in-game sens; the DPI normalizer rescales DPI instead
- Each `calc_sens_*` reads its game's FOV base(s) from `constants.fov.<game>` (`hipfire`/`base` — see root `CLAUDE.md:Change map → Add sens game`) but still hardcodes its own quirky sens-scale multiplier: `al`/`cs2` use `.022`, `bdo` uses `.001_84`, `fn` uses `.005_555`, `ow` uses raw rad multiplier `.006_6`, `r6` uses `.005_73`, `val` uses `.07`, `pubg` is log-scale (`base_sens=50`, `step≈15.0515`). Don't guess — copy from the game's actual config
- Sens is VIEW-INDEPENDENT — FPP/TPP is render-only. `state.camera.sens` is always plain `calc_rad_per_px(fov, width)` regardless of dimension, and every game match (hipfire/ADS/scope, TPP or FPP) is the plain formula at that view's FOV. The TPP camera-behind-pivot feel is re-created visually by the `state.camera.dist` orbit (`constants.tpp.<game>` × `render_dist_scale`), so parallax exists identically in trainer and game and cancels out of every sens number; yaw-per-count is the whole match. `constants.tpp.*` is therefore renderer-ONLY — tune orbit distance visually without touching sens. Feel-verified three times: pubg TPP-hipfire ↔ ADS both read plain and transition naturally in-game; bdo's old ×2.88 multiplier read ~133 at FHD when plain ≈46 was right; mc's old TPP parallax multiplier (`calc_tpp_rad_per_px`, ×~1.79 at vfov 110) read "way too slow" in-game (removed 2026-07 along with `calc_sens_fn_tpp`/`calc_sens_mc_tpp`/`calc_pubg_fpp_fov`/`constants.tpp.parallax_distance_scale`). Do NOT reintroduce a parallax/cross-view multiplier
- PUBG's `game.pubg_fov` preset is `80` or `103`. `80` uses the trainer's fixed-FOV TPP hipfire; `103` uses FPP hipfire at 103°. Only the Normal/hipfire calculation follows this preset because `FpsCameraFov` does not change the fixed ADS/scope FOVs. The INI rewrite writes the selected `FpsCameraFov`, and `calc_dpi_pubg()` applies the same FOV to both yaw scaling and monitor matching. The old `calc_pubg_fpp_fov()` bisection double-corrected for parallax that cancels — don't resurrect it

## When you change…
- A sens formula: also verify `logic.update_camera_view()` branch for the same `sens` key. The FOV the renderer uses must match the FOV the sens calc assumes; otherwise mouse and target scale diverge
- `calc_rad_per_px`: every `calc_sens_*` and `calc_dpi_*` calls it directly, but only two sites use it to populate the `state.camera.sens` cache — `logic.update_camera_view()` (fov change) + `render.resize_3d()` (width change). Readers (`render/camera.js:px_to_rad`/`rad_to_px`, `controller/window.js:on_mousemove`) read the cached `state.camera.sens`, so changing the formula flows to them automatically
- PUBG math: `calc_pubg.js` + `calc_sens_pubg()` + PUBG INI rewrite in `controller/game_sens.js` (all three)
- Add a sens game: see root `Change map → Add sens game`
